K8s 学习笔记一：容器及 Kubernetes 介绍

K8s 学习笔记

容器

容器的隔离机制：

  • Linux 命名空间，每个进程只能看到自己的系统视图（文件，进程，网络接口，主机名等）
  • Linux 控制组（cgroups），限制进程使用的资源量（CPU、内存、网络带宽等）

详细原理参看皓叔的 DOCKER基础技术：LINUX CGROUP DOCKER基础技术：LINUX NAMESPACE

Docker 容器平台

Docker 是一个打包分发运行应用程序的平台。

  • 镜像，包含了打包的应用程序及所依赖的环境。
  • 镜像仓库，集中存储镜像，可供上传下载镜像，有私有和公有仓库
  • 容器，Docker 基于给定的镜像创建的 Linux 容器，在宿主机上表现为一个或多个进程，并与宿主机环境隔离，同时资源使用也受到限制

Docker 镜像原理

  • 多层构成，不同镜像可能包含完全相同的层（比如都基于 ubuntu 系统镜像构建），提升镜像分发时的效率
  • 容器运行时是在镜像之上再创建一个读写层，所有写入操作都写进此层，若操作底层文件，则会在顶层创建此文件的拷贝

容器移植的限制

  • 因为容器共享宿主机内核，所以对应用运行时内核有要求的镜像可能不能运行在每台机器上。
  • 不同硬件架构的容器化应用也不能相互兼容，你不可能在 ARM 平台上运行 x86 架构的容器

其他容器运行时

比如基于 oci 标准的 containerd，用来作为 Docker 容器运行时的替代方案。

可以阅读这篇文章 终于可以像使用 Docker 一样丝滑地使用 Containerd 了

Kubernetes

简单概况就是一个字，帮助你编排管理成千上万的容器。

核心功能

  • 根据开发者提交的应用状态描述文件，自动部署到 k8s 集群，开发者和系统管理员无需关心部署在具体哪个节点上
  • 减少开发者对集群基础设施的维护 （ps. 就是集群学习成本有点高）
  • 提升整个基础设施的利用率

系统架构

  • 控制平面（Control Plane），控制和管理整个集群
  • 工作节点（Node），提供 k8s 运行环境，运行具体的应用

控制平面（Control Plane）

控制平面（Control Plane）对集群做出全局决策（比如调度），以及检测和响应集群事件。通常包括以下几个组件：

  • kube-apiserver 通信组件。
  • etcd 键值数据库，作为保存 k8s 集群所有数据的后台数据库。
  • kube-scheduler 调度组件，监视调度 pod。
  • kube-controller-manager 执行集群级别的功能，如复制节点、持续跟踪工作节点、处理节点失败等。

工作节点（Node）

工作节点（Node）运行容器化应用，包括以下几个组件：

  • kubelet 类似于一个 angent，与 kube-apiserver 通信，管理工作节点（Node）上的容器
  • kube-proxy 是每个节点上的网络代理，维护节点上的网络规则
  • Container Runtime 容器运行时，负责运行容器的软件。k8s 支持 Docker，containerd，CRI-O 以及任何实现 Kubernetes-CRI 标准的容器运行时软件。
